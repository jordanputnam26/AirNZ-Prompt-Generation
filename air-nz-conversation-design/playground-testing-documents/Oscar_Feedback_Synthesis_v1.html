This is a stack problem, not just a tone problem

Oscar is not mainly failing because the voice is far off. The stronger issue is that three things are being tuned at once:

  • the System Prompt layer that controls Oscar's voice and behaviour rules
  • the Service Agent block that tells Oscar what this turn is trying to do
  • the conversation-flow design choices about what to ask, when to confirm, when to hand over, and when to proactively move forward

In this prompt-engine setup, that third category is also being carried mostly by the Service Agent block. The System Prompt acts as the behaviour layer over the top.

That is why some messages sound almost right but still feel wrong in use. The wording is close, but the move underneath is too certain, too repetitive, badly timed, or not grounded in the actual capability.

Important tonal rule from Ben's review. Oscar should not aim for one fixed level of warmth. Its tone needs to flex by moment: softer when delivering a limit or handling frustration, and more direct, confident, and proactive when the path is clear and it should move the customer forward.

Detailed findings

What to carry into the next iteration

Each finding below combines the feedback call patterns with the exported JSON examples, then points to the most likely place to fix it.

Service Agent block System Prompt High priority

1. The rewriter scope is structurally mismatched to the scenarios being tested

What is happening

The rewriter prompt in the exported JSONs is explicitly framed as a Claim Missing Airpoints rewriter, but it is also being used for meal-request scenarios.

Why it matters

This makes the outputs harder to diagnose. Some misses look like voice problems when they are really architecture problems.

Recommended change

Either create separate rewriters for different service journeys, or use one rewriter that is genuinely generic and gets the journey type from the Service Agent block.

Service Agent block System Prompt High priority

2. Oscar is too certain too early in claim flows

Pattern

Oscar often behaves as if a claim is ready to proceed before enough information has been gathered or before prior customer action has been checked.

Reviewer signal

Terri consistently pushed for a more neutral opener: I can take a look, not we can get the claim underway.

Recommended change

Default to investigation language first, and add an explicit route for customers who may already have claimed.

System Prompt Service Agent block High priority

3. Repetition and over-confirmation are making Oscar feel effortful

Pattern

Oscar repeats reasons, repeats confirmation steps, re-lists details that were just provided, and sometimes re-greets mid-flow.

Reviewer signal

All three sessions surfaced this. Terri was comfortable with one confirmation per conversation in many cases, and Lauren flagged repeated handover reasoning as well.

Recommended change

Strengthen anti-repetition rules, especially around re-greeting, repeated bullets, and repeated reasoning after a boundary has already been made clear.

System Prompt High priority

4. Bad-news empathy is still miscalibrated

Pattern

Oscar reaches for phrases like disappointing, fair enough, or nuisance in ways that feel slightly off-brand, too human, or mildly dismissive.

Reviewer signal

Ben and Terri both pushed back on disappointing. Lauren was open to acknowledgement, but wanted it framed as I hear you, not emotional mirroring.

Recommended change

Move from generic empathy guidance to narrower moment-based rules: brief acknowledgement, plain reason, next valid step.

System Prompt Service Agent block High priority

4A. Ben's tonality point should be treated as an explicit design rule

Pattern

The system currently implies this balance, but does not name it clearly enough: Oscar should not sound the same in every moment.

What Ben was getting at

When the route is clear, Oscar should be confident, direct, and proactive. When the customer hits a limit, Oscar should stay warm enough not to sound harsh, patronising, or triggering.

Recommended change

Add an explicit calibrated-tonality rule to the voice system so Oscar is not tuned toward one averaged middle tone.

Service Agent block System Prompt Medium priority

5. Oscar sometimes explains the wrong thing, or too much of it

Pattern

The reply often includes details that are technically related but not the main answer. The gluten-free snack example is the clearest version of this.

Reviewer signal

Ben explicitly called this out as drift. Terri also wanted the domestic-meal explanation reframed around the simpler truth that there is no meal service on domestic flights.

Recommended change

Tighten the Service Agent block so the primary reason arrives clearly, and stop the System Prompt from expanding into adjacent detail unless it directly helps the current answer.

Service Agent block System Prompt Medium priority

6. Claim closings are too thin on expectation-setting

Pattern

Claim outcomes are often mechanically correct, but the close does not reduce uncertainty. Nothing else you need to do right now is not enough when the customer has already been waiting.

Reviewer signal

Terri repeatedly asked for clearer timing, next steps, and what the customer should expect if the result does not land in time.

Recommended change

Standardise a claim close pattern: what has been done, what happens next, likely timeframe if supported, and whether the customer needs to act again.

System Prompt Service Agent block Medium priority

7. Handover language is directionally right, but still too mechanical

Pattern

Oscar usually signals the right route, but the wording can still sound procedural, over-explained, or too choice-heavy.

Reviewer signal

Terri wanted more customer-relevant explanation. Lauren preferred softer path framing like here's what we could do for you rather than a hard two-option stance.

Recommended change

Create a handover pattern set by handover type, with clearer reason language and stronger continuity handling.

Service Agent block High priority

8. Privacy handling around other passengers needs a firmer rule

Pattern

The family meal flow surfaces passenger names before the customer has disclosed them in the conversation.

Reviewer signal

Terri landed on a safer pattern: confirm that there are multiple passengers on the booking, then ask the customer who the change is for.

Recommended change

Do not surface other passenger names unless the customer has already disclosed them, or the authority rule explicitly allows it.

Service Agent block Medium priority

9. Successful flows still create avoidable back and forth

Pattern

Even in positive journeys, Oscar often misses obvious proactive moves such as offering the return leg earlier or surfacing meal options sooner.

Reviewer signal

Lauren wanted a more proactive return-leg question. Terri also supported reducing back and forth where the next likely need is obvious.

Recommended change

Review successful service flows through the Service Agent block logic, not just the tone. Use visible booking context to reduce avoidable turns.

System Prompt Service Agent block Medium priority

10. Brand expression should be more deliberate, not more decorative

Pattern

The team does want more warmth and more Air New Zealand feel, but only in the right moments. The ask is not generic friendliness.

Reviewer signal

Terri supported controlled emoji and punctuation in upbeat closes. Lauren wanted more small brand moments in successful flows. Ben kept pulling the conversation back to confidence and momentum rather than decorative language.

Recommended change

Define where brand moments belong, especially in positive closes, and keep them out of frustrated or constrained moments.

Service Agent block System Prompt Medium priority

11. Oscar is sometimes promising help beyond the current supported scope

Pattern

Phrases like If there's anything else on the booking you want me to check, I can help with that are broader than the demonstrated capability.

Reviewer signal

Amy, Lauren, and Jared all pushed back on broad anything else language unless the experience can actually support it.

Recommended change

Narrow forward-help language to the real supported scope, or reserve it for handover moments where it is genuinely true.

Work split

Where the next work should sit

The sessions point to a fairly clean division of labour for the next iteration.

System Prompt

System Prompt work

  • Add an explicit calibrated-tonality rule so Oscar can flex between warmth and momentum by moment.
  • Remove fake-empathy defaults.
  • Tighten acknowledgement rules for bad news.
  • Strengthen anti-repetition and anti-re-greeting behaviour.
  • Stop expanding into adjacent explanation not asked for.
  • Narrow follow-on help language to supported scope.
  • Define positive close patterns separately from constrained closes.
Service Agent block

Service Agent block work

  • Fix the fact that the meal-request examples are still using a rewriter written for Claim Missing Airpoints.
  • Prevent early certainty in claim flows.
  • Improve expectation-setting fields for claim completion.
  • Make handoff reason and handoff type explicit in structured input.
  • Define privacy-safe passenger-name behaviour.
  • Pass enough prior-turn context for Oscar to avoid redundant confirmation.
  • Redesign success flows to reduce avoidable turns.
  • Decide where proactive next-step offers are beneficial.
  • Decide the tipping points for handover versus continued virtual handling.
Shared design decisions

Shared design decisions

These are still design decisions for the team to make, but in this implementation they will mostly be applied through Service Agent block changes or System Prompt changes.

  • Clarify where brand moments belong and where they should be suppressed.
  • Decide how broad Oscar is allowed to sound when offering more help.
  • Decide which moments need softness and which should prioritise direct forward motion.
Recommended sequence

Suggested order for the next round

This order keeps the team from over-tuning the voice before the underlying routing and scope issues are stable.

1

Resolve prompt setup first

Decide whether Oscar is using one general rewriter or separate rewriters for different service journeys.

2

Fix the highest-risk Service Agent block behaviours

Address early certainty in claims, privacy around passenger names, and over-broad help offers.

3

Retune the System Prompt against explicit rules

Focus on acknowledgement, repetition, bad-news phrasing, handover wording, and positive closes.

4

Rerun a smaller evidence set

Use one frustrated claim, one successful claim, one limitation flow, one successful SSR flow, and one privacy-driven handover.

Open decisions

What still needs product or brand input

  • Should Oscar apologise in constraint flows, and if so, in which exact moments?
  • Should Oscar ever surface other passenger names before the customer does?
  • What exact sign-off style is approved for upbeat travel moments?
  • Which emojis, if any, are approved for Oscar and in which contexts?
  • How broad is Oscar allowed to sound when offering further help?
  • At what frustration point should handover be offered rather than inferred later?
Bottom line

What this means for the team

The voice model is close enough to keep. The bigger problem is that it is currently being asked to compensate for the meal-request examples using the wrong rewriter, incomplete Service Agent block framing, weak handover design, and over-confirming interaction patterns.

In practical terms, most of the flow changes in this synthesis should be treated as Service Agent block changes, because that is where the turn strategy is being expressed in the current setup. The rest should be treated as System Prompt changes.

If those layers are separated and tightened in order, the next round of Oscar outputs should improve quickly and be much easier to judge cleanly.
